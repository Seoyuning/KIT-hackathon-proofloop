# Design: ProofLoop

2026 Korea IT Academy Vibe Coding Hackathon 기획 문서
Mode: Builder fast-track
Status: MVP 구현 기준

## 문제 정의

공교육과 학원 수업 현장에서 학생의 질문과 교사의 수업 준비는 서로 단절되어 있습니다.

- 학생은 궁금한 내용을 즉시 물어볼 곳이 마땅치 않고, 교과서 범위를 벗어난 답을 받는 경우가 많습니다.
- 교사는 한 반 전체가 어디서 막히는지 한눈에 보기 어렵고, 같은 질문을 수업마다 개별적으로 다시 다룹니다.
- 수업 자료와 시험지를 만들 때, 실제로 학생이 어려워한 지점 대신 교사의 감에 의존하게 됩니다.

이건 "더 나은 AI 튜터" 문제가 아니라, **학생 질문과 교사 수업 준비를 같은 교과서 데이터 위에서 연결하는 문제**입니다.

## 왜 지금인가

- 공교육과 학원 모두 교재 기반 수업을 여전히 핵심으로 사용합니다.
- 생성형 AI는 확산됐지만, 교과서 범위에 정확히 고정된 학습 도구는 드뭅니다.
- 학생이 AI에 자유롭게 질문하는 흐름과 교사가 그 데이터를 수업 설계에 활용하는 흐름이 같은 제품 안에 연결된 사례는 거의 없습니다.

## 타깃 사용자와 가장 좁은 진입점

### 주요 사용자

- **학생**: 교과서 단원 범위 안에서 질문하고 근거가 있는 답을 받고 싶은 학습자
- **교사**: 학생 질문 패턴을 보고 수업 자료와 시험 초안을 빠르게 준비하고 싶은 선생님

### 가장 좁은 진입점

한 반의 학생이 교과서 단원 범위 안에서 챗봇에게 질문하고, 교사가 그 질문 DB를 바탕으로 수업 자료 한 장을 만드는 순간.

## 제품 개요

ProofLoop는 교과서 기반 학습 스튜디오입니다. 학생과 교사는 하나의 앱 안에서 다른 화면을 보지만, **같은 교과서와 같은 질문 DB**를 공유합니다.

### 공통

- 이메일 + 비밀번호 기반 인증
- 회원가입 시 역할 선택 (학생 / 교사)
- 역할에 따라 접근 가능한 페이지가 완전히 분리됨

### 학생 플로우 (`/studio/chat`)

1. 학년 / 과목 / 출판사 기준으로 교과서 봇을 선택합니다.
2. 선택한 교과서 범위 안에서만 답하는 챗봇과 대화합니다.
3. 답변은 단원 이름과 페이지 근거를 함께 보여줍니다.
4. 학생이 던진 질문은 자동으로 통합 질문 DB에 적립됩니다.

### 교사 플로우 (`/studio/analysis`, `/studio/generate`)

1. `/studio/analysis`에서 학생 질문 DB를 단원별로 정렬된 클러스터 형태로 확인합니다.
2. 어떤 단원에서 질문이 몰리는지, 어떤 개념이 반복되는지 한눈에 파악합니다.
3. `/studio/generate`에서 선택한 범위를 기준으로 수업 자료 초안과 시험지 초안을 생성합니다.

## 핵심 설계 원칙

- **교과서에 고정된 답변**: 학생 답변은 교과서 범위를 벗어나지 않도록 설계되며, 단원·페이지 근거가 함께 노출됩니다.
- **같은 데이터로 연결된 두 화면**: 학생의 질문은 교사의 분석/생성 입력이 되고, 교사의 생성은 학생이 배우는 범위 위에서 이루어집니다.
- **역할 경계의 명확화**: 학생은 교사용 페이지를 볼 수 없고, 그 반대도 마찬가지입니다. 혼란을 줄이고 각 사용자가 필요한 화면에 바로 도달합니다.

## 라우트 요약

| 라우트 | 역할 | 목적 |
|--------|------|------|
| `/` | 공통 | `/studio`로 리다이렉트 |
| `/studio` | 공통 | 인증 상태에 따라 역할별 페이지로 자동 이동 |
| `/studio/login` | 공통 | 이메일+비밀번호 로그인 / 회원가입 + 역할 선택 |
| `/studio/chat` | 학생 | 교과서 기반 챗봇 |
| `/studio/analysis` | 교사 | 질문 DB 분석 + 교과서 범위 개요 |
| `/studio/generate` | 교사 | 수업 자료 / 시험 초안 생성 |

## 시드 데이터

MVP 단계에서는 다음 교과서 봇 데이터가 시드되어 있습니다.

- 고등학교 수학
- 중학교 과학
- 고등학교 국어

실제 교과서 PDF 업로드와 파싱은 차후 단계의 작업이며, 현재는 로컬 소스 파일 기반 시뮬레이션입니다.

## 기술 구성

- **Next.js 16 / React 19 / TypeScript / Tailwind CSS 4**
- **Auth**: `src/lib/auth-context.tsx` — localStorage 기반 MVP, 역할 고정
- **Shared state**: `src/lib/studio-context.tsx` — 봇 선택, 채팅, 질문 DB, 교사 생성 결과 공유
- **생성 로직**: `src/lib/studio-generation.ts` — 교과서 범위 답변, 질문 클러스터링, 수업자료/시험 초안 생성
- **시드 데이터**: `src/lib/studio-data.ts` — 교과서 봇, 단원, 초기 질문 클러스터

## 고려한 대안

### 대안 A: 범용 AI 튜터

교과서 범위를 벗어난 답이 쉽게 나오고, 교사 측 활용 흐름이 끊어짐 → 제외

### 대안 B: 교사용 수업자료 생성기 단독

학생 질문이라는 실데이터를 받지 못해 "AI가 상상한 반" 기준으로만 자료를 만들게 됨 → 제외

### 대안 C: ProofLoop (채택)

학생 챗봇과 교사 분석/생성이 같은 교과서와 같은 질문 DB로 연결되므로, 수업 준비가 학생의 실제 어려움에 정렬됨

## 현재 한계

- 교과서 근거는 시드된 로컬 데이터 기반 시뮬레이션
- 질문 DB는 인메모리/로컬 소스 기반 MVP (실제 DB 아님)
- 교사 생성 결과는 결정론적 로컬 로직 (실시간 LLM 파이프라인 아님)
- 인증은 localStorage 기반 MVP (서버 세션 아님)
