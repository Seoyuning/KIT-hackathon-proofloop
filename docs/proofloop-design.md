# Design: ProofLoop

2026 Korea IT Academy Vibe Coding Hackathon 기획 문서
Status: 구현 완료 기준

## 문제 정의

공교육과 학원 수업 현장에서 세 가지 문제가 동시에 존재합니다.

1. **학생의 AI 과제 대행**: 학생이 AI를 과제 대행 수단으로 사용하면 학습이 일어나지 않습니다. AI 사용을 금지하면 학습 도구로서의 가치도 사라집니다.
2. **학생 개인별 약점 파악 불가**: 학생마다 이해가 부족한 개념이 다른데, 이를 데이터로 파악할 수 있는 도구가 없습니다. 범용 AI는 교과서 범위를 벗어나거나 근거가 불명확합니다.
3. **학생-교사 데이터 단절**: 교사는 학생 한 명 한 명의 약점과 반 전체 공통 난점을 동시에 봐야 하지만, 이를 데이터로 확인할 경로가 없어 수업 자료와 시험지를 감에 의존해 만듭니다.

이건 "더 나은 AI 튜터" 문제가 아니라, **AI 사용 자체가 학습 행위가 되면서 동시에 교사에게 데이터를 전달하는 구조** 문제입니다.

## 핵심 아이디어: 지속적 학습 검증 루프

학생은 AI를 자유롭게 사용할 수 있지만, AI가 학습을 대신해 줄 수는 없습니다.

**모든 대화에서** AI는 다음 세 가지를 동시에, 매번 수행합니다:

1. **교과서 근거 답변** — 해당 반의 교과서 범위 안에서만 답하며, 단원명과 쪽수 근거를 포함
2. **이해도 실시간 측정** — 매 답변에서 학생의 이해 수준을 1(매우 부족)~5(우수)로 판정, DB에 저장
3. **후속 서술형 질문** — 매 답변 끝에 찍어서 맞출 수 없는 서술형 질문을 제시

과제를 AI에게 시키면? AI가 답을 거절하고 핵심 개념 설명 + 서술형 확인 질문을 발행합니다. 학생이 답하면 그 답변을 다시 분석해 이해 수준을 재평가합니다. **AI 사용 자체가 학습 행위**가 되는 역전 구조입니다.

## 타깃 사용자

### 학생
- 교과서 범위 안에서 근거 있는 답을 받고 싶은 학습자
- AI를 자유롭게 쓰되, 진짜 이해를 확인받고 싶은 학생

### 교사
- 학생별 약점과 반 전체 공통 난점을 데이터로 파악하고 싶은 선생님
- 학생 데이터에 정렬된 수업 자료와 시험지를 빠르게 만들고 싶은 선생님

### 가장 좁은 진입점

교사가 반을 만들고 초대 코드를 학생에게 공유 → 학생이 챗봇으로 질문 → AI가 검증하며 데이터 저장 → 교사가 대시보드에서 확인하고 수업 자료를 생성하는 순간.

## 제품 구조

### 공통

- Supabase 기반 이메일+비밀번호 인증 (서버 프록시 방식)
- 회원가입 시 역할 선택 (학생/교사) + 학생은 학년 선택 (중1~고3)
- 역할에 따라 접근 가능한 페이지가 완전히 분리됨

### 학생 플로우

1. 가입 시 학년(중1~고3)을 선택합니다.
2. 사이드바에서 교사가 공유한 6자리 초대 코드를 입력해 반에 참여합니다.
3. 반에 참여하면 해당 반의 교과서(학년+과목+출판사)에 맞는 챗봇이 자동 활성화됩니다.
4. 챗봇과 대화하면 매 답변에 교과서 근거 + 이해도 측정 + 후속 서술형 질문이 함께 나옵니다.
5. 과제를 시키면 AI가 답을 거절하고 학생을 역으로 검증합니다.
6. 모든 질문과 이해도 데이터가 Supabase DB에 자동 저장됩니다.

### 교사 플로우

1. `/studio/classes`에서 반을 만듭니다 (학년, 과목, 출판사, 교과서 지정). 6자리 초대 코드가 자동 생성되며 클릭하면 복사됩니다.
2. `/studio/analysis`에서 실시간 대시보드를 확인합니다: 누적 질문 수, 참여 학생 수, 상위 오개념, 평균 이해도(5점 만점). "반 전체 공통 약점"과 "학생별 약점" 두 뷰를 전환합니다.
3. `/studio/classes`에서 학생별 약점 트래킹: 각 학생의 질문 횟수, 약점 개념, 오개념 태그, 최근 질문을 확인합니다.
4. `/studio/generate`에서 단원을 선택하고 수업 자료 초안(슬라이드 아웃라인, 확인 질문, 교사 메모)과 시험 초안(4지선다 문항+정답+근거+예상 함정)을 생성합니다.
5. 결과물을 클립보드 복사 또는 파일 다운로드로 즉시 내보냅니다.

## 핵심 설계 원칙

- **지속적 검증**: 일반 질문이든 과제 질문이든 예외 없이 이해도를 측정하고 서술형 질문을 발행합니다. AI를 아무리 써도 진짜 이해 없이 빠져나갈 수 없습니다.
- **교과서 범위 고정**: 모든 답변은 교과서 단원/쪽수 근거를 포함하며, 범위 밖 질문에는 명확하게 경계를 설정합니다.
- **같은 데이터, 두 가지 뷰**: 학생의 질문 데이터가 교사에게 개인별 약점 + 반 공통 오개념 + 이해도 수치로 전달됩니다. 중복 입력 없이 개인화와 집계가 동시에 일어납니다.
- **역할 경계 명확화**: 학생은 교사용 페이지를 볼 수 없고 그 반대도 마찬가지입니다.

## 데이터 구조

### Supabase 테이블

- **profiles**: id, email, name, role, grade, subject, created_at
- **classes**: id, teacher_id, name, subject, grade, publisher, textbook_name, invite_code, max_students
- **class_members**: id, class_id, student_id, joined_at
- **student_questions**: id, class_id, student_id, question, section_title, misconception, understanding_level, created_at

### 교과서 카탈로그

`src/lib/textbook-catalog.ts`에 55종 실제 한국 교과서(10개 출판사, 15개 과목)를 정적 배열로 관리. 런타임 API 호출 없이 즉시 사용.

## 기술 구성

- **Next.js 16 / React 19 / TypeScript / Tailwind CSS 4**
- **인증**: Supabase Auth, 서버 프록시 방식 (`/api/auth/*` Route Handlers)
- **DB**: Supabase PostgreSQL + Row Level Security
- **AI**: Gemini 2.5 Flash (학생 챗봇, `/api/chat`)
- **상태 관리**: `AuthProvider`(인증) + `StudioProvider`(도메인) 분리
- **교사 생성 로직**: `src/lib/studio-generation.ts` — 결정론적, 외부 API 없이 항상 동작
- **교과서 데이터**: `src/lib/textbook-catalog.ts` (55종 카탈로그) + `src/lib/studio-data.ts` (시드 단원)

## AI 활용 구조

### Gemini 2.5 Flash — 학생 챗봇 + 검증기

- System instruction에 교과서 단원 데이터(제목, 쪽수, 요약, 핵심어, 오개념 태그, 인용 기준) 전량 주입
- 매 답변에서 `[근거]`, `[이해도]`, `[후속 질문]` 태그 강제 출력
- "풀어줘/답 알려줘" 감지 시 답 거절 + 개념 설명 + 서술형 확인 질문
- 최근 10개 대화를 context로 전달해 맥락 유지
- 응답 파싱 후 Supabase `student_questions` 테이블에 자동 저장

### Claude Code — 개발 에이전트

- 기획 정제부터 아키텍처, 프론트엔드, API, DB 스키마, 문서화까지 일관된 품질로 수행
- 사람이 방향과 도메인 판단을, AI가 속도와 구현력을 담당하는 협업 구조

## 고려한 대안

| 대안 | 거절 이유 |
|---|---|
| 범용 AI 튜터 | 교과서 범위 이탈 + 교사 측 연결 불가 + 과제 대행 문제 해결 불가 |
| 교사용 생성기 단독 | 학생 실데이터 없이 "AI가 상상한 반" 기준 자료만 생성 |
| AI 사용 금지 | 학습 도구로서의 가치까지 제거 |
| 별도 과제 대행 감지 미들웨어 | 추가 API 비용+지연. system prompt 내장이 더 간결 |
| 이해도 별도 API 호출 | 비용 2배. 하나의 Gemini 호출에서 동시 추출이 효율적 |
