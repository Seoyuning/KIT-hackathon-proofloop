# AI 빌딩 리포트 최종 초안

아래 내용은 해커톤 제출 양식에 바로 옮겨 적을 수 있도록 정리한 최종본입니다.
팀명, 대표자 이름, 휴대폰번호만 채우면 됩니다.

## 프로젝트명

ProofLoop

## 한 줄 소개

ProofLoop는 학생이 교과서 범위 안에서 질문하고, 교사는 그 질문 데이터를 바탕으로 수업 자료와 시험 초안을 만드는, 교과서 기반 학생-교사 연결형 학습 스튜디오입니다.

## 1. 기획

### 설정한 사용자는 누구이며, 해결하려는 구체적인 문제점/불편함은 무엇인가요?

주 사용자는 공교육 및 학원 수업의 **학생**과 **교사**입니다.

현재 교육 현장에는 두 가지 단절이 있습니다.

- 학생이 교과서 내용을 물어볼 곳은 많지만, 교과서 범위에 정확히 고정된 답을 주는 도구는 드뭅니다. 대부분의 AI는 범위를 벗어나거나 출처가 불명확합니다.
- 교사는 반 전체가 어디서 막혔는지 데이터로 확인하기 어렵고, 수업 자료와 시험지를 만들 때 감에 의존하게 됩니다.

ProofLoop는 이 두 흐름을 같은 교과서와 같은 질문 DB 위에서 연결합니다. 학생이 묻는 질문이 곧 교사가 수업을 준비하는 입력이 됩니다.

### 문제를 해결하기 위한 솔루션의 핵심 기능은 무엇인가요?

1. **교과서 기반 챗봇 (학생)**
   학년·과목·출판사 기준으로 교과서 봇을 선택하면, 해당 교과서 범위 안에서만 답하고 단원·페이지 근거를 함께 보여줍니다.

2. **통합 질문 DB**
   학생이 던진 질문은 자동으로 통합 DB에 적립되고, 단원별로 클러스터링되어 "어디서 질문이 몰리는지"를 보여줍니다.

3. **교사 질문 분석 (교사)**
   교사는 자신이 가르치는 범위의 질문 클러스터를 보고, 학생들이 실제로 어려워하는 지점을 정렬된 형태로 확인합니다.

4. **수업 자료·시험 초안 생성 (교사)**
   선택한 교과서 범위와 질문 데이터를 기준으로 수업 자료 초안과 시험 초안을 즉시 생성합니다.

5. **역할 기반 접근**
   회원가입 시 학생/교사 역할을 선택하고, 이후 각 역할은 자신에게 필요한 페이지만 접근합니다.

### 이 솔루션이 도입되었을 때 기대되는 개선점이 무엇인가요?

- 학생은 교과서 범위 안에서 근거 있는 답을 즉시 받을 수 있습니다.
- 교사는 반 전체의 학습 어려움을 데이터로 먼저 보고, 수업을 그 지점 위주로 재구성할 수 있습니다.
- 수업 자료와 시험지 준비 시간을 크게 줄이면서도, 실제 학생 데이터에 정렬된 결과물을 만들 수 있습니다.
- 공교육과 학원 모두에서, 하나의 도구로 학생-교사 플로우를 연결할 수 있습니다.

## AI 활용 전략

### 이 프로젝트에서 사용할 AI 도구와 모델은 무엇이며, 선택한 이유는 무엇인가요?

- **Claude Code 기반 개발 에이전트**
  기획 정제, 아키텍처 설계, Next.js 프론트엔드 구현, 문서화까지 일관된 품질로 빠르게 밀어붙이기 위해 사용했습니다.

- **로컬 생성 로직 (`src/lib/studio-generation.ts`)**
  교과서 범위 기반 답변, 질문 클러스터링, 수업 자료 및 시험 초안 생성을 결정론적으로 수행해, 심사/데모 환경에서 외부 API 상태와 무관하게 항상 동작하도록 설계했습니다.

- **시드 데이터 (`src/lib/studio-data.ts`)**
  고등학교 수학, 중학교 과학, 고등학교 국어 교과서 데이터를 시드해 교과서 기반 플로우가 즉시 시연 가능하도록 했습니다.

### 각 AI 도구별 활용 전략을 작성해주세요.

#### 1. Claude Code 개발 에이전트

- **페르소나**
  교육 도메인을 이해하고 제품 기획과 구현을 동시에 끌고 가는 시니어 풀스택 엔지니어

- **핵심 전략**
  "AI 튜터"나 "AI 채점기" 같은 포화된 카테고리가 아니라, 공교육/학원 현장에서 이미 발생하는 **교과서 범위 고정**과 **학생-교사 데이터 단절** 문제를 좁게 정의하는 것부터 시작했습니다. 그 다음 학생 챗봇과 교사 분석/생성을 같은 교과서 데이터 위에 올리는 구조를 설계했습니다.

- **산출물**
  `/studio` 라우트 세트, 역할 기반 인증/접근 제어, 공유 Context, 교과서 봇 시드 데이터, 학생/교사 페이지, 문서

#### 2. 로컬 생성 로직 (Deterministic generator)

- **페르소나**
  교과서 범위를 엄격히 지키는 학습 보조 엔진

- **핵심 전략**
  생성 결과가 항상 교과서 단원/페이지 근거를 함께 반환하도록 구조화했습니다. 학생 답변, 질문 클러스터, 수업 자료 초안, 시험 초안은 모두 같은 교과서 데이터 스키마 위에서 만들어집니다.

- **역할 분리**
  - 학생용: 교과서 범위 기반 질의응답
  - 교사용: 질문 DB 클러스터링 + 자료/시험 초안 생성

#### 3. 시드 데이터 기반 MVP

- **페르소나**
  심사·데모 환경을 위한 재현 가능한 교실

- **핵심 전략**
  실제 PDF 업로드 파이프라인은 범위에서 제외하고, 대표 교과(수학/과학/국어) 시드만으로도 학생 질문 → 교사 분석 → 자료 생성 흐름 전체를 시연 가능하게 만들었습니다.

### 토큰 낭비를 최소화하고 유지보수성 및 재현성을 높이기 위한 전략이 있다면 작성해주세요.

- 생성 로직을 `src/lib/studio-generation.ts` 한 곳에 중앙 집중화해, 학생 답변/교사 생성의 규칙을 한 파일에서 관리하도록 했습니다.
- 상태 관리는 `StudioProvider`와 `AuthProvider`로 분리해, 학생-교사 공유 데이터(교과서, 질문 DB)와 인증을 명확히 구분했습니다.
- 역할별 페이지를 완전히 분리(`/studio/chat`, `/studio/analysis`, `/studio/generate`)해 조건부 렌더링 분기를 줄이고 유지보수를 단순화했습니다.
- 시드 데이터는 `src/lib/studio-data.ts`에 고정해, 데모와 테스트 재현성을 확보했습니다.
- 공개 저장소에 비밀키가 들어가지 않도록 환경변수는 `.env.example` 형태로만 관리합니다.

## 심사 포인트 대응 요약

### 기술적 완성도

- 이메일+비밀번호 인증, 역할 기반 접근 제어, 역할별 페이지 완전 분리까지 구현했습니다.
- 학생 챗봇과 교사 분석/생성이 동일한 교과서 데이터 스키마와 질문 DB를 공유합니다.
- `npx next build`가 통과하는 배포 가능한 웹앱 상태입니다.

### AI 활용 능력 및 효율성

- AI를 "더 많이 답하는 튜터"가 아니라 **교과서 범위 해석기 + 학생 데이터 정렬기**로 배치했습니다.
- 학생의 질문이 그대로 교사 수업 준비의 입력이 되는 구조를 만들었습니다.

### 기획력 및 실무 접합성

- 공교육 및 학원 수업에서 이미 존재하는 "교과서 고정 + 학생-교사 데이터 연결" 문제를 겨냥했습니다.
- 학생, 교사 모두 자신의 화면에서 즉시 체감 가능한 가치가 있도록 설계했습니다.

### 창의성

- 기존 AI 교육 제품이 "학생을 위한 튜터" 또는 "교사를 위한 생성기"로 나뉘어 있었다면, ProofLoop는 두 사용자를 같은 교과서 데이터 위에서 연결합니다.

## 2. AI 협업 방식 기록

이 섹션은 "ProofLoop가 AI와 얼마나, 어떻게 협력해서 만들어졌는가"를 심사위원이 한눈에 확인할 수 있도록 구성했습니다. 결과물만이 아니라 **과정**과 **판단**을 함께 기록합니다.

### 2.1 사람–AI 역할 분담

| 영역 | 사람(팀) | AI(Claude Code) | 공동 |
|---|---|---|---|
| 제품 방향·포지셔닝 | ✅ 최종 결정 | 대안 제시 | 기획 토론 |
| 타겟 사용자 정의 | ✅ 현장 지식 기반 확정 | 후보 페르소나 생성 | 용어 다듬기 |
| 라우트 및 정보 구조 | 검토/승인 | ✅ 초안 설계 | 이름 정리 |
| UI 컴포넌트 구현 | 검토·수동 조정 | ✅ 초안 작성 | 스타일 토큰 합의 |
| 인증/역할 게이트 | 요구사항 정의 | ✅ Supabase 통합 구현 | 엣지 케이스 검토 |
| 시드 데이터 큐레이션 | ✅ 과목/범위 선정 | 교과서 구조 템플릿화 | - |
| 문서/README/리포트 | 톤·강조점 결정 | ✅ 초안 작성 | 퇴고 |
| 디버깅 | 재현/증상 보고 | ✅ 원인 분석 및 수정안 | 회귀 테스트 |

> 핵심 원칙: **AI는 "빠른 초안과 구현력", 사람은 "방향 결정과 도메인 판단"**을 담당합니다. AI 제안을 그대로 수용하지 않고 매 단계 사람이 채택/거절/수정했습니다.

### 2.2 의사결정 로그 (채택/거절/수정)

| 주제 | AI 1차 제안 | 사람의 결정 | 결정 이유 |
|---|---|---|---|
| 초기 제품 형태 | "AI 튜터 + AI 채점기" 조합 | ❌ 거절 | 포화 카테고리. 교과서 범위 고정 + 학생–교사 연결이라는 좁은 문제로 재정의 |
| 랜딩 노출 전략 | 쿠키 게이트로 비공개 프리뷰 운영 | ❌ 거절 (되돌림) | 공모전에서는 공개 랜딩이 필수. `/`에서 바로 포지셔닝이 보이게 변경 |
| 학생/교사 한 페이지 통합 | 한 페이지에서 역할 토글 | ❌ 거절 | 조건부 렌더링 복잡도 증가. 역할별 페이지 완전 분리 (`/studio/chat`, `/studio/analysis`, `/studio/generate`) |
| 상태 관리 방식 | 전역 단일 Context | ⚠️ 수정 채택 | `StudioProvider`(도메인)와 `AuthProvider`(인증)로 분리 |
| 생성 로직 구현 위치 | 페이지 내부 분산 | ❌ 거절 | `src/lib/studio-generation.ts` 단일 파일로 중앙화 (재현성·유지보수) |
| 인증 스택 | localStorage MVP만 유지 | ✅ 초안 채택 후 업그레이드 | MVP는 localStorage, 이후 Supabase 연동으로 전환 (`feat: supabase auth...` 커밋) |
| 교과서 시드 범위 | 과목 전체 | ❌ 거절 | 심사·데모 재현성을 위해 수학·과학·국어 3과목으로 한정 |
| 한국어/영어 혼용 | 영어 기본 | ❌ 거절 | 현장 사용자 기준 한국어 우선 (`feat: localize UI for Korean-first demo`) |

### 2.3 AI 제안을 **거절**한 사례 (진정한 협업의 증거)

심사 관점에서 가장 중요한 부분입니다. AI를 맹목적으로 따라간 프로젝트가 아니라는 것을 보여주는 구체적 기록입니다.

1. **"게이트로 숨겨진 랜딩" 되돌리기**
   - AI가 쿠키 기반 프리뷰 구조(`/preview`, `/preview/landing`, `LANDING_PREVIEW_KEY`)를 제안하고 구현했지만, 공모전 공개 원칙과 맞지 않아 **전부 제거**하고 `/`를 퍼블릭 랜딩으로 복구했습니다.
   - 근거: `src/app/page.tsx`는 현재 공개 랜딩, `src/app/preview/*` 삭제, README에서 해당 섹션 제거.

2. **"역할 토글 단일 페이지" 거절**
   - AI는 처음에 한 페이지에서 student/teacher를 토글하는 구조를 제안했지만, 역할별 정보 밀도와 보안 경계가 달라 **페이지 자체를 분리**했습니다.
   - 근거: `feat: split studio into role-based pages (student/teacher)` 커밋.

3. **"생성 로직을 페이지에 직접 작성" 거절**
   - AI 초안은 각 페이지에서 답변/클러스터/자료 초안을 바로 만들었지만, 재사용성과 테스트 가능성을 위해 `src/lib/studio-generation.ts`로 리팩터링을 요청했습니다.

### 2.4 AI 제안을 **채택**한 사례 (효율성의 증거)

- **라우트 구조 초안**: `/studio` 아래 chat/analysis/generate 레이아웃 제안을 그대로 채택. 사람이 같은 구조를 설계하는 데 들였을 수 시간을 절감.
- **인증 리팩터링 경로**: localStorage MVP → Supabase로 단계적 이전 경로를 AI가 제안, 그대로 진행. (`b28ca5f feat: switch auth to email + password`, `5fcdc48 feat: supabase auth, mypage, splash landing, and keepalive`)
- **스타일 토큰 중앙화**: `paper-panel`, `eyebrow`, `navy/orange` 등 재사용 토큰을 AI가 제안, 전체 페이지에서 일관되게 사용.

### 2.6 일일 협업 로그

`report/` 디렉터리에 날짜별 협업 기록을 유지합니다.

- `report/20260409_report.md`, `report/20260409_resume.txt`
- `report/20260410_report.md`, `report/20260410_resume.txt`

각 리포트에는 그날의 목표, AI에게 요청한 프롬프트 요지, AI가 만든 산출물, 사람이 수정/거절한 부분이 시간순으로 기록되어 있습니다. 심사위원은 이 리포트를 통해 하루 단위로 협업 궤적을 확인할 수 있습니다.

### 2.7 프롬프트 운영 원칙 (재현성)

1. **작은 단위로 맡기기** — 한 번에 여러 페이지를 바꾸지 않고, 라우트 하나 / 컴포넌트 하나 단위로 요청.
2. **근거 포함 요구** — 교과서 답변은 반드시 단원/페이지 근거를 함께 반환하도록 AI에 강제.
3. **문서와 코드 동기화** — 기능 변경 시 README와 리포트 갱신을 같은 요청에 묶어 드리프트 방지.
4. **검증 루프** — `npm run lint`, `npm run build`, 로컬 UI 확인을 AI 작업 직후 사람이 수행.
5. **거절 로그 유지** — 이 문서 2.2~2.3 섹션이 그 결과물입니다.

### 2.8 한 줄 요약
