# AI 빌딩 리포트

## 프로젝트명

ProofLoop

## 한 줄 소개

ProofLoop는 Gemini 2.5 Flash AI 챗봇이 학생을 매 대화마다 검증하고, 그 데이터가 실시간으로 교사에게 흘러가 수업이 되는, 교과서 기반 개인화 학습 루프입니다.

## 1. 기획

### 설정한 사용자는 누구이며, 해결하려는 구체적인 문제점/불편함은 무엇인가요?

주 사용자는 공교육 및 학원 수업의 **학생**과 **교사**입니다.

현재 교육 현장에는 세 가지 문제가 동시에 존재합니다.

- **학생 쪽**: 학생마다 이해가 부족한 개념이 다른데, 이를 스스로 정확히 파악하고 교과서 범위 안에서 보완할 수 있는 개인화된 도구가 거의 없습니다. 범용 AI는 범위를 벗어나거나 근거가 불명확해서 학습에 그대로 쓰기 어렵습니다. 또한 학생이 AI를 과제 대행 수단으로 사용하면 오히려 학습을 방해합니다.
- **교사 쪽**: 교사는 학생 한 명 한 명의 약점과, 반 전체가 공통적으로 어려워하는 지점을 동시에 파악해야 하지만, 이를 데이터로 확인할 수 있는 경로가 거의 없어 수업 자료와 시험지를 만들 때 감에 의존하게 됩니다.

ProofLoop는 이 두 흐름을 **같은 교과서와 같은 반(class) 시스템 위에서 하나의 학습 루프로** 연결합니다. 학생은 AI와 학습하면서 자신의 약점을 파악하고, 그 과정에서 쌓인 데이터는 그대로 교사에게 **개인별 약점 + 반 공통 오개념 + 단원별 이해도**로 전달됩니다.

### 문제를 해결하기 위한 솔루션의 핵심 기능은 무엇인가요?

1. **Gemini 2.5 Flash 기반 교과서 챗봇 + 지속적 학습 검증 루프 (학생)**
   학생은 AI 챗봇을 자유롭게 사용할 수 있지만, AI가 학습을 대신해 줄 수는 없습니다. 모든 대화에서 AI는 다음 세 가지를 **동시에, 매번** 수행합니다:
   - **교과서 근거 답변**: 해당 반의 교과서 범위 안에서만 답하며, 모든 답변에 단원명과 쪽수 근거를 포함합니다.
   - **이해도 실시간 측정**: 매 답변에서 학생의 이해 수준을 1(매우 부족)~5(우수)로 판정합니다. 이 데이터는 DB에 저장되어 교사 대시보드에서 단원별로 집계됩니다.
   - **후속 서술형 질문**: 매 답변 끝에 찍어서 맞출 수 없는 서술형 질문을 제시해, 학생이 자기 말로 설명해야만 다음 단계로 넘어갈 수 있도록 유도합니다.
   이 세 가지가 일반 질문에도 과제 질문에도 항상 작동하기 때문에, 학생이 AI를 아무리 많이 사용해도 **진짜 이해 없이 빠져나갈 수 없는 구조**입니다.

2. **과제 대행 감지 강화**
   학생이 "이 문제 풀어줘", "답 알려줘" 등 답을 직접 요구하면 AI는 답 자체를 거절하고 핵심 개념을 설명한 뒤 서술형 확인 질문을 발행합니다. 과제를 AI에게 시킬 수는 있지만, AI가 오히려 학생을 검증하는 역전이 일어납니다.

3. **반(Class) 시스템과 6자리 초대 코드**
   교사가 반을 만들면 자동으로 6자리 영문+숫자 초대 코드가 생성됩니다. 학생은 이 코드를 입력해 반에 참여하며, 반당 최대 35명까지 수용합니다. 교사는 반 생성 시 학년/과목/출판사/교과서를 지정하고, 여러 반을 동시에 관리할 수 있습니다.

4. **55종 이상 실제 한국 교과서 카탈로그**
   비상교육, 미래엔, 천재교육, 동아출판, 교학사, 좋은책신사고, YBM 등 10개 출판사의 15개 과목, 55종 이상의 실제 교과서 데이터를 탑재했습니다.

5. **교사 실시간 대시보드**
   누적 질문 수, 참여 학생 수, 상위 오개념을 한눈에 확인합니다. 단원별 평균 이해도를 집계해 어떤 단원에서 학생들이 막히는지 데이터로 보여줍니다.

6. **학생별 약점 트래킹 (per-student)**
   반 관리 페이지에서 각 학생의 질문 횟수, 약점 개념 목록, 오개념 태그, 최근 질문을 실시간으로 확인합니다.

7. **수업 자료/시험 초안 생성 + 내보내기**
   교사가 단원을 선택하고 수업 목표/시간을 설정하면 강의 자료를 생성합니다. 시험지는 4지선다 문항 + 정답 + 근거 + 예상 함정을 생성합니다. 클립보드 복사 또는 파일 다운로드로 즉시 내보냅니다.

8. **교사 학습 자료 업로드**
   교과서 범위 외에 교사 자체 학습지/자료를 사진 촬영, 사진 선택, 파일 업로드로 추가할 수 있습니다.

9. **서버 프록시 인증 + 보안 강화**
   Supabase 인증은 서버 사이드에서 처리. RLS 정책으로 학생 질문 데이터는 본인과 담당 교사만 접근 가능. 보안 헤더(X-Frame-Options, X-Content-Type-Options 등) 적용. 오픈 리다이렉트 차단.

10. **반응형 UI + 모바일 대응**
    Pretendard 폰트 기반 한국어 최적화 UI, 모바일 햄버거 메뉴, `word-break: keep-all`로 한국어 줄바꿈 최적화.

### 이 솔루션이 도입되었을 때 기대되는 개선점이 무엇인가요?

- **학생 측 개인화 + 지속적 검증**: 학생은 AI를 자유롭게 사용하되, 매 답변에서 이해도 측정 + 후속 서술형 질문이 따라오기 때문에 "AI한테 시키기만 하는" 사용이 구조적으로 불가능합니다. 과제를 AI에게 맡기려 해도 AI가 오히려 학생을 검증하므로, AI 사용 자체가 학습 행위가 됩니다.
- **교사 측 데이터 기반 수업**: 교사는 같은 데이터에서 (1) 학생 개개인의 약점, (2) 반 전체 공통 오개념, (3) 단원별 이해도를 동시에 확인할 수 있어, 개별 지도와 수업 설계를 같은 신호 위에서 정합시킬 수 있습니다.
- **운영 즉시성**: 교사가 반을 만들고 초대 코드를 학생에게 공유하면, 5분 안에 전체 루프가 작동하기 시작합니다.
- **플로우 연결**: 공교육과 학원 모두에서, 하나의 도구로 학생 개인화 학습과 교사의 수업 준비를 같은 루프로 이을 수 있습니다.

## AI 활용 전략

### 이 프로젝트에서 사용할 AI 도구와 모델은 무엇이며, 선택한 이유는 무엇인가요?

- **Gemini 2.5 Flash (학생 챗봇 엔진 + 학습 검증기)**
  학생과의 실시간 대화에 사용합니다. 빠른 응답 속도와 한국어 능력이 뛰어나며, 교과서 데이터를 system instruction으로 주입해 grounded answering을 구현합니다. 핵심은 단순 답변이 아니라 **매 대화마다 교과서 근거 + 이해도 측정 + 서술형 후속 질문을 동시에 수행하는 지속적 검증 루프**를 하나의 API 호출로 처리한다는 점입니다.

- **Claude Code (개발 에이전트)**
  기획 정제, 아키텍처 설계, Next.js 프론트엔드 구현, API 라우트 설계, Supabase 스키마 통합, 보안 강화, 문서화까지 일관된 품질로 빠르게 수행하기 위해 사용했습니다.

- **결정론적 생성 로직 (`src/lib/studio-generation.ts`)**
  교사용 강의 자료 및 시험 초안 생성을 결정론적으로 수행해, 심사/데모 환경에서 외부 API 상태와 무관하게 항상 동작하도록 설계했습니다.

### 각 AI 도구별 활용 전략을 작성해주세요.

#### 1. Gemini 2.5 Flash — 학생 학습 챗봇

- **페르소나**
  교과서 범위를 엄격히 지키는 학습 코치이자 지속적 학습 검증기

- **핵심 전략 — 지속적 검증 루프**
  - System instruction에 교과서 단원 데이터(제목, 쪽수, 요약, 핵심어, 오개념 태그, 인용 기준)를 전량 주입
  - 범위 밖 질문에는 "이 교과서 범위에서는 다루지 않는 내용입니다"로 명확하게 경계 설정
  - **매 답변마다** `[근거] 단원/쪽수`, `[이해도] 1~5`, `[후속 질문]` 태그를 강제 — 일반 질문이든 과제 질문이든 예외 없이 이해도를 측정하고 서술형 질문을 발행
  - "풀어줘/답 알려줘" 패턴을 감지하면 답 자체를 거절하고 개념 설명 + 서술형 확인 질문 발행
  - 최근 10개 대화를 context로 전달해 대화 맥락 유지

- **데이터 파이프라인**
  학생 질문 → Gemini 응답(답변+근거+이해도+후속질문) → Supabase `student_questions` 테이블에 저장(class_id, student_id, question, section_title, misconception, understanding_level) → 교사 대시보드에서 실시간 집계

#### 2. Claude Code — 개발 에이전트

- **페르소나**
  교육 도메인을 이해하고 제품 기획과 구현을 동시에 끌고 가는 시니어 풀스택 엔지니어

- **핵심 전략**
  "AI 튜터"나 "AI 채점기" 같은 포화된 카테고리가 아니라, 공교육/학원 현장에서 이미 발생하는 **교과서 범위 고정**과 **학생-교사 데이터 단절** 문제를 좁게 정의하는 것부터 시작했습니다.

- **산출물**
  반 시스템(classes, class_members, student_questions 스키마), 6자리 초대 코드 생성/참여 API, Gemini 챗봇 라우트, 실시간 대시보드 API, 55종 교과서 카탈로그, 역할 기반 인증/접근 제어, 보안 강화(RLS, 인증 체크, 보안 헤더), 내보내기 기능, 반응형 UI, 문서

#### 3. 결정론적 생성 로직 (Lesson/Exam generator)

- **페르소나**
  교과서 범위를 엄격히 지키는 교사용 수업 보조 엔진

- **핵심 전략**
  생성 결과가 항상 교과서 단원/페이지 근거를 함께 반환하도록 구조화했습니다. 강의 자료(슬라이드 아웃라인, 확인 질문, 교사 메모)와 시험 초안(4지선다 문항, 정답, 근거, 예상 함정)은 모두 같은 교과서 데이터 스키마 위에서 만들어집니다. 클립보드 복사와 `.txt` 파일 다운로드를 모두 지원합니다.

### 토큰 낭비를 최소화하고 유지보수성 및 재현성을 높이기 위한 전략이 있다면 작성해주세요.

- **Gemini 호출 최적화**: system instruction에 교과서 데이터를 정적으로 주입하고, 대화 히스토리는 최근 10개만 전달해 토큰 효율성을 확보합니다. temperature 0.7로 일관된 답변 품질을 유지합니다.
- **구조화된 출력 강제**: `[근거]`, `[이해도]`, `[후속 질문]` 태그를 system prompt에서 강제해, 후처리 파싱이 정확하게 동작합니다.
- **서버 사이드 DB 저장**: 학생 질문과 이해도를 매 대화마다 자동 저장하되, 실패해도 응답은 반환해 UX를 해치지 않습니다.
- **교사용 생성 로직 중앙화**: `src/lib/studio-generation.ts` 한 곳에 강의/시험 생성 규칙을 관리해, 외부 API 없이도 항상 동작합니다.
- **상태 관리 분리**: `StudioProvider`(도메인)와 `AuthProvider`(인증)로 분리해, 학생-교사 공유 데이터와 인증을 명확히 구분했습니다.
- **역할별 페이지 완전 분리**: `/studio/chat`, `/studio/analysis`, `/studio/generate`, `/studio/classes`로 분리해 조건부 렌더링 분기를 줄이고 유지보수를 단순화했습니다.
- **교과서 카탈로그 정적 관리**: `src/lib/textbook-catalog.ts`에 55종 교과서를 정적 배열로 관리해, 런타임 API 호출 없이 즉시 사용 가능합니다.

## 심사 포인트 대응 요약

### 기술적 완성도

- Supabase 기반 이메일+비밀번호 인증, 서버 프록시 방식으로 보안 강화
- 반 시스템: 6자리 초대 코드 생성, 최대 35명, 교사 다중 반 관리
- Gemini 2.5 Flash API 연동: 교과서 grounded answering, 과제 대행 감지, 이해도 평가
- 실시간 교사 대시보드: 누적 질문, 참여 학생, 단원별 이해도, 학생별 약점
- RLS 정책으로 학생 데이터 보호, 보안 헤더 적용, API 인증 체크

### AI 활용 능력 및 효율성

- AI를 "더 많이 답하는 튜터"가 아니라 **매 대화마다 학생을 검증하는 지속적 학습 검증기**로 배치했습니다.
- AI를 쓸수록 학습이 일어나는 역전 구조: 과제를 AI에게 맡겨도 AI가 학생을 역으로 검증합니다.
- 하나의 질문 DB를 두 사용자(학생, 교사)가 서로 다른 뷰로 소비하는 구조입니다.

### 기획력 및 실무 접합성

- 교사가 반을 만들고 초대 코드를 공유하면 즉시 시작되는 low-friction 온보딩
- 55종 실제 교과서를 탑재해 "내 교과서가 없다"는 현장 장벽을 사전에 제거
- 수업 자료 생성 결과를 클립보드 복사/파일 다운로드로 즉시 활용 가능

### 창의성

- 기존 AI 교육 제품이 "학생을 위한 튜터" 또는 "교사를 위한 생성기"로 나뉘어 있었다면, ProofLoop는 **학생의 개인화 학습**과 **교사의 집계 분석**을 같은 반 시스템과 교과서 데이터 루프 위에서 동시에 만들어 냅니다.
- AI 사용을 막는 대신, AI가 매 대화에서 학생을 역으로 검증하는 구조로 "AI는 학습을 방해한다"는 교사의 우려를 기능으로 해소합니다.

## 2. AI 협업 방식 기록

이 섹션은 "ProofLoop가 AI와 얼마나, 어떻게 협력해서 만들어졌는가"를 심사위원이 한눈에 확인할 수 있도록 구성했습니다.

### 2.1 사람-AI 역할 분담

| 영역 | 사람(팀) | AI(Claude Code) | 공동 |
|---|---|---|---|
| 제품 방향/포지셔닝 | 최종 결정 | 대안 제시 | 기획 토론 |
| 타겟 사용자 정의 | 현장 지식 기반 확정 | 후보 페르소나 생성 | 용어 다듬기 |
| 반 시스템 설계 | 요구사항(초대코드, 35명 제한) | 스키마+API 구현 | 플로우 검증 |
| Gemini 프롬프트 설계 | 과제 대행 감지 요구 | system prompt 작성 | 출력 포맷 합의 |
| UI 컴포넌트 구현 | 검토/수동 조정 | 초안 작성 | 스타일 토큰 합의 |
| 보안 강화 | 보안 감사 요구 | RLS/인증/헤더 구현 | 취약점 검토 |
| 문서/README/리포트 | 톤/강조점 결정 | 초안 작성 | 퇴고 |

> 핵심 원칙: **AI는 "빠른 초안과 구현력", 사람은 "방향 결정과 도메인 판단"**을 담당합니다.

### 2.2 의사결정 로그 (채택/거절/수정)

| 주제 | AI 1차 제안 | 사람의 결정 | 결정 이유 |
|---|---|---|---|
| 초기 제품 형태 | "AI 튜터 + AI 채점기" 조합 | 거절 | 포화 카테고리. 교과서 범위 고정 + 학생-교사 연결로 재정의 |
| 챗봇 모델 선택 | OpenAI GPT-4o | 거절 | Gemini 2.5 Flash가 한국어+빠른 응답에 적합 |
| 과제 대행 대응 | 별도 필터링 미들웨어 | 수정 채택 | system prompt에 직접 감지 규칙 삽입이 더 간결 |
| 이해도 평가 | 별도 평가 API 호출 | 거절 | 하나의 Gemini 호출에서 동시 추출 (비용 절감) |
| 평균 이해도 표시 | 전체 평균 이해도 카드 | 거절 | 챕터별 이해도가 의미 있음. 전체 평균은 제거 |
| 반 시스템 | 교사가 학생을 직접 등록 | 거절 | 초대 코드 방식이 현장에서 더 자연스러움 |
| 교과서 데이터 | PDF 업로드 파이프라인 | 거절 | 해커톤 기간 내 불가. 향후 출판사 API 연동 계획 |
| 인증 스택 | 클라이언트에서 직접 Supabase 호출 | 수정 채택 | 서버 프록시 방식으로 변경해 보안 강화 |

### 2.3 AI 제안을 거절한 사례

1. **"과제 대행을 별도 미들웨어로 감지" 거절** — 추가 API 비용과 지연. system prompt 안에 직접 감지 규칙 삽입이 더 간결.
2. **"이해도 평가 별도 API" 거절** — 비용 2배. 하나의 Gemini 호출에서 동시 추출.
3. **"PDF 업로드로 교과서 데이터 추출" 거절** — 해커톤 3일 내 완성 불가. 정적 카탈로그로 전환.
4. **"전체 평균 이해도 표시" 거절** — 챕터마다 학생 이해도가 다른데 전체 평균은 무의미. 단원별 집계로 변경.
5. **"게이트로 숨겨진 랜딩" 되돌리기** — 공모전 공개 원칙과 맞지 않아 전부 제거.
