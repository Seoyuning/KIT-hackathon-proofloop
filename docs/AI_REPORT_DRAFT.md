# AI 빌딩 리포트 최종 초안

아래 내용은 해커톤 제출 양식에 바로 옮겨 적을 수 있도록 정리한 최종본입니다.
팀명, 대표자 이름, 휴대폰번호만 채우면 됩니다.

## 프로젝트명

ProofLoop

## 한 줄 소개

ProofLoop는 Gemini 2.5 Flash AI 챗봇으로 학생이 교과서 범위 안에서 학습하면서 자신의 약점을 파악하고, 교사는 반 시스템과 실시간 대시보드로 학생 개개인의 약점과 반 공통 오개념을 동시에 확인해 수업 자료와 시험 초안을 만드는, 교과서 기반 개인화 학습 루프입니다.

## 1. 기획

### 설정한 사용자는 누구이며, 해결하려는 구체적인 문제점/불편함은 무엇인가요?

주 사용자는 공교육 및 학원 수업의 **학생**과 **교사**입니다.

현재 교육 현장에는 두 가지 단절이 있습니다.

- **학생 쪽**: 학생마다 이해가 부족한 개념이 다른데, 이를 스스로 정확히 파악하고 교과서 범위 안에서 보완할 수 있는 개인화된 도구가 거의 없습니다. 범용 AI는 범위를 벗어나거나 근거가 불명확해서 학습에 그대로 쓰기 어렵습니다. 또한 학생이 AI를 과제 대행 수단으로 사용하면 오히려 학습을 방해합니다.
- **교사 쪽**: 교사는 학생 한 명 한 명의 약점과, 반 전체가 공통적으로 어려워하는 지점을 동시에 파악해야 하지만, 이를 데이터로 확인할 수 있는 경로가 거의 없어 수업 자료와 시험지를 만들 때 감에 의존하게 됩니다.

ProofLoop는 이 두 흐름을 **같은 교과서와 같은 반(class) 시스템 위에서 하나의 학습 루프로** 연결합니다. 학생은 AI와 학습하면서 자신의 약점을 스스로 파악하고, 그 과정에서 쌓인 데이터는 그대로 교사에게 **개인별 약점 + 반 공통 오개념 + 5단계 이해도**라는 세 가지 뷰로 전달됩니다.

### 문제를 해결하기 위한 솔루션의 핵심 기능은 무엇인가요?

1. **Gemini 2.5 Flash 기반 교과서 챗봇 (학생)**
   학생이 반에 참여하면 해당 반의 교과서 범위 안에서만 답하는 AI 챗봇과 대화합니다. 모든 답변에 단원명과 쪽수 근거가 포함됩니다. AI는 학생의 이해 수준을 5단계로 실시간 평가하고, 후속 서술형 질문을 제시해 깊은 학습을 유도합니다.

2. **과제 대행 감지 및 학습 유도**
   학생이 "이 문제 풀어줘", "답 알려줘" 등 답을 직접 요구하면, AI는 바로 답을 주지 않고 핵심 개념을 설명한 뒤 서술형 확인 질문(찍어서 맞출 수 없는 형태)을 내어 스스로 풀도록 유도합니다.

3. **5단계 이해도 평가 시스템**
   매 대화에서 AI가 학생의 이해 수준을 1(매우 부족)~5(우수)로 판단합니다. 이 데이터는 DB에 저장되어 교사 대시보드에서 학생별·반 전체 평균 이해도로 집계됩니다.

4. **반(Class) 시스템과 6자리 초대 코드**
   교사가 반을 만들면 자동으로 6자리 영문+숫자 초대 코드가 생성됩니다. 학생은 이 코드를 입력해 반에 참여하며, 반당 최대 35명까지 수용합니다. 교사는 반 생성 시 학년·과목·출판사·교과서를 지정합니다.

5. **55종 이상 실제 한국 교과서 카탈로그**
   비상교육, 미래엔, 천재교육, 동아출판, 교학사, 좋은책신사고, YBM 등 10개 출판사의 수학·국어·영어·과학·사회·한국사·물리학·화학·생명과학·지구과학·정보·기술가정·도덕·음악·미술 등 55종 이상의 실제 교과서 데이터를 탑재했습니다.

6. **교사 분석 대시보드 (실시간 데이터)**
   교사는 `/api/dashboard`에서 집계된 실시간 데이터로 누적 질문 수, 참여 학생 수, 상위 오개념, 평균 이해도(5점 만점)를 한눈에 확인합니다. "반 전체 공통 약점" 탭과 "학생별 약점" 탭을 전환하며 개별 지도와 수업 설계를 같은 신호 위에서 진행합니다.

7. **학생별 약점 트래킹 (per-student)**
   반 관리 페이지에서 각 학생의 질문 횟수, 약점 개념 목록, 오개념 태그, 최근 질문을 실시간으로 확인합니다. 같은 단원에서 반복 질문이 발생하면 자동으로 약점으로 집계됩니다.

8. **수업 자료·시험 초안 생성 + 내보내기**
   교사가 단원을 선택하고 수업 목표/시간을 설정하면 강의 자료(슬라이드 아웃라인, 확인 질문, 교사 메모)를 생성합니다. 시험지는 문항 수(1~10), 시험 목적을 설정해 4지선다 문항 + 정답 + 근거 + 예상 함정을 생성합니다. 결과물은 **클립보드 복사** 또는 **파일 다운로드**로 즉시 내보낼 수 있습니다.

9. **서버 프록시 인증 (Server-proxied Auth)**
   모든 Supabase 인증 호출은 서버 사이드에서 처리됩니다. 클라이언트에 anon key가 노출되지만 실제 DB 접근은 서버 Route Handler를 통해 이루어져 보안을 강화했습니다.

10. **반응형 UI + 모바일 대응**
    Pretendard 폰트 기반 한국어 최적화 UI, 모바일에서는 햄버거 메뉴로 사이드바를 토글합니다. `word-break: keep-all`로 한국어가 음절 중간에서 끊기지 않도록 처리했습니다.

11. **Keepalive로 Supabase 슬립 방지**
    무료 Supabase 인스턴스가 비활성으로 꺼지는 것을 막기 위해 주기적 keepalive 요청을 구현했습니다.

### 이 솔루션이 도입되었을 때 기대되는 개선점이 무엇인가요?

- **학생 측 개인화 + 올바른 학습 습관**: 학생은 교과서 범위 안에서 근거 있는 답을 받고, 과제 대행을 시도하면 AI가 이를 감지해 서술형 질문으로 전환합니다. 스스로 답을 구성하는 과정에서 진짜 이해도가 드러납니다.
- **교사 측 삼중 뷰**: 교사는 같은 데이터에서 (1) 학생 개개인의 약점, (2) 반 전체 공통 오개념, (3) 평균 이해도 수치를 동시에 확인할 수 있어, 개별 지도와 수업 설계를 같은 신호 위에서 정합시킬 수 있습니다.
- **정렬된 자료 생성**: 수업 자료와 시험지 준비 시간을 크게 줄이면서도, 실제 학생 데이터에 정렬된 결과물을 만들고 바로 내보낼 수 있습니다.
- **운영 즉시성**: 교사가 반을 만들고 초대 코드를 학생에게 공유하면, 5분 안에 전체 루프가 작동하기 시작합니다. 별도 설정이나 PDF 업로드 없이 55종 교과서에서 바로 시작합니다.
- **플로우 연결**: 공교육과 학원 모두에서, 하나의 도구로 학생 개인화 학습과 교사의 수업 준비를 같은 루프로 이을 수 있습니다.

## AI 활용 전략

### 이 프로젝트에서 사용할 AI 도구와 모델은 무엇이며, 선택한 이유는 무엇인가요?

- **Gemini 2.5 Flash (학생 챗봇 엔진)**
  학생과의 실시간 대화에 사용합니다. 빠른 응답 속도와 한국어 능력이 뛰어나며, 교과서 데이터를 system instruction으로 주입해 grounded answering을 구현합니다. 과제 대행 감지, 5단계 이해도 평가, 후속 질문 생성까지 하나의 API 호출로 처리합니다.

- **Claude Code (개발 에이전트)**
  기획 정제, 아키텍처 설계, Next.js 프론트엔드 구현, API 라우트 설계, Supabase 스키마 통합, 문서화까지 일관된 품질로 빠르게 밀어붙이기 위해 사용했습니다.

- **결정론적 생성 로직 (`src/lib/studio-generation.ts`)**
  교사용 강의 자료 및 시험 초안 생성을 결정론적으로 수행해, 심사/데모 환경에서 외부 API 상태와 무관하게 항상 동작하도록 설계했습니다.

### 각 AI 도구별 활용 전략을 작성해주세요.

#### 1. Gemini 2.5 Flash — 학생 학습 챗봇

- **페르소나**
  교과서 범위를 엄격히 지키는 학습 코치이자 과제 대행 감시자

- **핵심 전략**
  - System instruction에 교과서 단원 데이터(제목, 쪽수, 요약, 핵심어, 오개념 태그, 인용 기준)를 전량 주입
  - 범위 밖 질문에는 "이 교과서 범위에서는 다루지 않는 내용입니다"로 명확하게 경계 설정
  - "풀어줘/답 알려줘" 패턴을 감지하면 답을 주지 않고 개념 설명 + 서술형 확인 질문 발행
  - 매 답변 끝에 `[근거] 단원/쪽수`, `[이해도] 1~5`, `[후속 질문]` 태그를 강제해 구조화된 출력 확보
  - 최근 10개 대화를 context로 전달해 대화 맥락 유지

- **데이터 파이프라인**
  학생 질문 → Gemini 응답(답변+근거+이해도+후속질문) → Supabase `student_questions` 테이블에 저장(class_id, student_id, question, section_title, misconception, understanding_level) → 교사 대시보드에서 실시간 집계

#### 2. Claude Code — 개발 에이전트

- **페르소나**
  교육 도메인을 이해하고 제품 기획과 구현을 동시에 끌고 가는 시니어 풀스택 엔지니어

- **핵심 전략**
  "AI 튜터"나 "AI 채점기" 같은 포화된 카테고리가 아니라, 공교육/학원 현장에서 이미 발생하는 **교과서 범위 고정**과 **학생-교사 데이터 단절** 문제를 좁게 정의하는 것부터 시작했습니다. 그 다음 학생 챗봇과 교사 분석/생성을 같은 교과서 데이터 위에 올리는 구조를 설계했습니다.

- **산출물**
  반 시스템(classes, class_members, student_questions 스키마), 6자리 초대 코드 생성/참여 API, Gemini 챗봇 라우트, 실시간 대시보드 API, 55종 교과서 카탈로그, 역할 기반 인증/접근 제어, 내보내기 기능, 반응형 UI, 문서

#### 3. 결정론적 생성 로직 (Lesson/Exam generator)

- **페르소나**
  교과서 범위를 엄격히 지키는 교사용 수업 보조 엔진

- **핵심 전략**
  생성 결과가 항상 교과서 단원/페이지 근거를 함께 반환하도록 구조화했습니다. 강의 자료(슬라이드 아웃라인, 확인 질문, 교사 메모)와 시험 초안(4지선다 문항, 정답, 근거, 예상 함정)은 모두 같은 교과서 데이터 스키마 위에서 만들어집니다.

- **내보내기**
  클립보드 복사와 `.txt` 파일 다운로드를 모두 지원해, 교사가 바로 문서 편집기에 붙여넣거나 인쇄할 수 있습니다.

### 토큰 낭비를 최소화하고 유지보수성 및 재현성을 높이기 위한 전략이 있다면 작성해주세요.

- **Gemini 호출 최적화**: system instruction에 교과서 데이터를 정적으로 주입하고, 대화 히스토리는 최근 10개만 전달해 토큰 효율성을 확보합니다. temperature 0.7로 일관된 답변 품질을 유지합니다.
- **구조화된 출력 강제**: `[근거]`, `[이해도]`, `[후속 질문]` 태그를 system prompt에서 강제해, 후처리 파싱이 정확하게 동작합니다.
- **서버 사이드 DB 저장**: 학생 질문과 이해도를 매 대화마다 자동 저장하되, 실패해도 응답은 반환해 UX를 해치지 않습니다.
- **교사용 생성 로직 중앙화**: `src/lib/studio-generation.ts` 한 곳에 강의/시험 생성 규칙을 관리해, 외부 API 없이도 항상 동작합니다.
- **상태 관리 분리**: `StudioProvider`(도메인)와 `AuthProvider`(인증)로 분리해, 학생-교사 공유 데이터(교과서, 질문 DB)와 인증을 명확히 구분했습니다.
- **역할별 페이지 완전 분리**: `/studio/chat`, `/studio/analysis`, `/studio/generate`, `/studio/classes`로 분리해 조건부 렌더링 분기를 줄이고 유지보수를 단순화했습니다.
- **교과서 카탈로그 정적 관리**: `src/lib/textbook-catalog.ts`에 55종 교과서를 정적 배열로 관리해, 런타임 API 호출 없이 즉시 사용 가능합니다.
- **환경변수 분리**: 공개 저장소에 비밀키가 들어가지 않도록 환경변수를 `.env.example` 형태로만 관리합니다.

## 심사 포인트 대응 요약

### 기술적 완성도

- Supabase 기반 이메일+비밀번호 인증, 서버 프록시 방식으로 보안 강화
- 반 시스템: 6자리 초대 코드 생성, 최대 35명 수용, 학생 참여/탈퇴
- Gemini 2.5 Flash API 연동: 교과서 grounded answering, 과제 대행 감지, 5단계 이해도 평가
- 실시간 교사 대시보드: 누적 질문, 참여 학생, 상위 오개념, 평균 이해도 집계
- 수업 자료/시험 초안 생성 + 클립보드 복사/파일 다운로드 내보내기
- 55종 실제 한국 교과서 카탈로그 (10개 출판사, 15개 과목)
- 반응형 UI: Pretendard 폰트, 모바일 햄버거 메뉴, 한국어 줄바꿈 최적화
- `npx next build`가 통과하는 배포 가능한 웹앱 상태

### AI 활용 능력 및 효율성

- AI를 "더 많이 답하는 튜터"가 아니라 **학습 코치 + 과제 대행 감시자 + 이해도 측정기**로 배치했습니다.
- 학생이 답을 요구하면 AI가 거절하고 서술형 질문을 내는 구조로, AI가 학습을 대체하지 않고 촉진합니다.
- 매 대화의 이해도(1~5)가 DB에 저장되어, 교사는 숫자로 학생 상태를 파악합니다.
- 하나의 질문 DB를 두 사용자(학생 개인, 교사)가 서로 다른 뷰로 소비하는 구조로 설계해, 중복 입력 없이 개인화와 집계가 동시에 일어납니다.
- Gemini API 호출은 학생 챗봇 한 곳에만 집중하고, 교사용 생성은 결정론적 로직으로 처리해 비용을 최소화합니다.

### 기획력 및 실무 접합성

- 공교육 및 학원 수업에서 이미 존재하는 "학생 개인별 약점 추적 + 반 전체 공통 약점 파악" 문제를 겨냥했습니다.
- 교사가 반을 만들고 초대 코드를 공유하면 즉시 시작되는 low-friction 온보딩을 구현했습니다.
- 55종 실제 교과서를 탑재해 "내 교과서가 없다"는 현장 장벽을 사전에 제거했습니다.
- 과제 대행 감지를 넣어, 학교/학원에서 "학생이 AI로 숙제를 대행한다"는 현실적 우려에 대응합니다.
- 수업 자료 생성 결과를 클립보드 복사/파일 다운로드로 즉시 활용할 수 있어, 기존 워크플로우에 바로 삽입됩니다.

### 창의성

- 기존 AI 교육 제품이 "학생을 위한 튜터" 또는 "교사를 위한 생성기"로 나뉘어 있었다면, ProofLoop는 **학생의 개인화 학습**과 **교사의 집계 분석**을 같은 반 시스템과 교과서 데이터 루프 위에서 동시에 만들어 냅니다.
- 과제 대행 감지를 AI 챗봇에 내장해, "AI는 학습을 방해한다"는 교사의 우려를 기능으로 해소합니다.
- 이해도를 숫자로 정량화해 교사 대시보드에 띄움으로써, "이 학생이 얼마나 이해하고 있는가"를 감이 아닌 데이터로 판단하게 합니다.

## 2. AI 협업 방식 기록

이 섹션은 "ProofLoop가 AI와 얼마나, 어떻게 협력해서 만들어졌는가"를 심사위원이 한눈에 확인할 수 있도록 구성했습니다. 결과물만이 아니라 **과정**과 **판단**을 함께 기록합니다.

### 2.1 사람–AI 역할 분담

| 영역 | 사람(팀) | AI(Claude Code) | 공동 |
|---|---|---|---|
| 제품 방향·포지셔닝 | 최종 결정 | 대안 제시 | 기획 토론 |
| 타겟 사용자 정의 | 현장 지식 기반 확정 | 후보 페르소나 생성 | 용어 다듬기 |
| 반 시스템 설계 | 요구사항(초대코드, 35명 제한) | 스키마+API 구현 | 플로우 검증 |
| Gemini 프롬프트 설계 | 과제 대행 감지 요구 | system prompt 작성 | 출력 포맷 합의 |
| UI 컴포넌트 구현 | 검토·수동 조정 | 초안 작성 | 스타일 토큰 합의 |
| 인증/역할 게이트 | 서버 프록시 방식 결정 | Supabase 통합 구현 | 엣지 케이스 검토 |
| 교과서 카탈로그 | 과목/출판사/교과서 선정 | 55종 데이터 구조화 | 검수 |
| 문서/README/리포트 | 톤·강조점 결정 | 초안 작성 | 퇴고 |
| 디버깅 | 재현/증상 보고 | 원인 분석 및 수정안 | 회귀 테스트 |

> 핵심 원칙: **AI는 "빠른 초안과 구현력", 사람은 "방향 결정과 도메인 판단"**을 담당합니다. AI 제안을 그대로 수용하지 않고 매 단계 사람이 채택/거절/수정했습니다.

### 2.2 의사결정 로그 (채택/거절/수정)

| 주제 | AI 1차 제안 | 사람의 결정 | 결정 이유 |
|---|---|---|---|
| 초기 제품 형태 | "AI 튜터 + AI 채점기" 조합 | 거절 | 포화 카테고리. 교과서 범위 고정 + 학생-교사 연결이라는 좁은 문제로 재정의 |
| 챗봇 모델 선택 | OpenAI GPT-4o 사용 | 거절 | 비용 대비 속도. Gemini 2.5 Flash가 한국어+빠른 응답에 적합 |
| 과제 대행 대응 | 별도 필터링 미들웨어 | 수정 채택 | system prompt에 직접 감지 규칙 삽입이 더 간결하고 유지보수 용이 |
| 이해도 평가 | 별도 평가 API 호출 | 거절 | 한 번의 Gemini 호출에서 답변+근거+이해도+후속질문을 동시에 추출 (비용 절감) |
| 반 시스템 | 교사가 학생을 직접 등록 | 거절 | 초대 코드 방식이 현장에서 더 자연스러움 (카카오톡으로 코드 공유) |
| 학생/교사 한 페이지 통합 | 한 페이지에서 역할 토글 | 거절 | 조건부 렌더링 복잡도 증가. 역할별 페이지 완전 분리 |
| 교과서 데이터 | PDF 업로드 파이프라인 구축 | 거절 | 해커톤 기간 내 완성 불가. 55종 카탈로그 정적 탑재로 전환 |
| 인증 스택 | 클라이언트에서 직접 Supabase 호출 | 수정 채택 | 서버 프록시 방식으로 변경해 보안 강화 |
| 상태 관리 방식 | 전역 단일 Context | 수정 채택 | `StudioProvider`(도메인)와 `AuthProvider`(인증)로 분리 |
| 한국어/영어 혼용 | 영어 기본 | 거절 | 현장 사용자 기준 한국어 우선 |
| 내보내기 기능 | PDF 생성 라이브러리 도입 | 수정 채택 | 의존성 최소화 위해 클립보드 복사 + txt 다운로드로 단순화 |

### 2.3 AI 제안을 **거절**한 사례 (진정한 협업의 증거)

심사 관점에서 가장 중요한 부분입니다. AI를 맹목적으로 따라간 프로젝트가 아니라는 것을 보여주는 구체적 기록입니다.

1. **"과제 대행을 별도 미들웨어로 감지" 거절**
   - AI가 별도 분류 모델/미들웨어를 제안했지만, 추가 API 호출 비용과 지연을 고려해 Gemini system prompt 안에 감지 규칙을 직접 삽입하는 방식을 선택했습니다. 단일 호출로 감지+응답+평가가 동시에 이루어집니다.

2. **"이해도 평가 별도 API" 거절**
   - AI가 답변 후 별도로 이해도를 판정하는 2단계 호출을 제안했지만, 비용과 지연을 절반으로 줄이기 위해 하나의 Gemini 응답에서 `[이해도] 1~5` 태그를 강제 출력하도록 구조화했습니다.

3. **"PDF 업로드로 교과서 데이터 추출" 거절**
   - AI가 OCR + 벡터 DB 파이프라인을 제안했지만, 해커톤 3일 내에 안정적으로 완성하기 어렵다고 판단해 55종 교과서를 정적 카탈로그로 탑재하고, 단원 구조는 시드 데이터로 제공하는 방식을 선택했습니다.

4. **"역할 토글 단일 페이지" 거절**
   - AI는 처음에 한 페이지에서 student/teacher를 토글하는 구조를 제안했지만, 역할별 정보 밀도와 보안 경계가 달라 **페이지 자체를 분리**했습니다.

5. **"게이트로 숨겨진 랜딩" 되돌리기**
   - AI가 쿠키 기반 프리뷰 구조를 제안하고 구현했지만, 공모전 공개 원칙과 맞지 않아 **전부 제거**하고 `/`를 퍼블릭 스플래시로 복구했습니다.

### 2.4 AI 제안을 **채택**한 사례 (효율성의 증거)

- **Gemini system prompt 구조**: 교과서 단원 데이터를 `### 단원명 (쪽수)` 형식으로 구조화하는 방안을 AI가 제안, 그대로 채택. 출력 태그(`[근거]`, `[이해도]`, `[후속 질문]`) 형식도 AI 초안을 활용.
- **6자리 초대 코드 생성 로직**: 혼동 가능 문자(0/O, 1/I/L)를 제외한 영문+숫자 32자 풀에서 랜덤 생성 + 충돌 검증 재시도 로직을 AI가 제안, 그대로 채택.
- **대시보드 API 구조**: 교사의 모든 반에 걸쳐 질문/학생/오개념/이해도를 집계하는 단일 엔드포인트를 AI가 제안, 프론트엔드 호출 횟수를 최소화.
- **라우트 구조**: `/studio` 아래 chat/analysis/generate/classes/mypage 분리를 AI가 제안, 그대로 채택.
- **인증 리팩터링 경로**: localStorage MVP → Supabase → 서버 프록시로 단계적 이전 경로를 AI가 제안, 그대로 진행.
- **내보내기 UX**: 클립보드 복사 + 파일 다운로드 이중 버튼 패턴을 AI가 제안, 교사 편의성 극대화.

### 2.7 프롬프트 운영 원칙 (재현성)

1. **작은 단위로 맡기기** — 한 번에 여러 페이지를 바꾸지 않고, 라우트 하나 / 컴포넌트 하나 단위로 요청.
2. **근거 포함 강제** — Gemini 답변은 반드시 단원/페이지 근거 + 이해도 + 후속 질문을 포함하도록 system prompt에서 강제.
3. **문서와 코드 동기화** — 기능 변경 시 README와 리포트 갱신을 같은 요청에 묶어 드리프트 방지.
4. **검증 루프** — `npm run lint`, `npm run build`, 로컬 UI 확인을 AI 작업 직후 사람이 수행.
5. **거절 로그 유지** — 이 문서 2.2~2.3 섹션이 그 결과물입니다.
6. **데이터 파이프라인 검증** — 학생 질문 → DB 저장 → 대시보드 집계 흐름을 매 변경 시 E2E로 확인.
